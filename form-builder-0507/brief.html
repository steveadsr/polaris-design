Form Builder — Colleague Brief

What this feature does, how to navigate the mockup, and open questions to discuss.

↗ What Is the Form Builder?

Plugin developers using the Polaris licensing platform can build an activation form that appears inside their JUCE plugin when a user activates their license. The form collects user data (name, email, machine info) and stores it against the license record in Polaris.

Each form is product-scoped — Surge XT has its own form, Syna-X has its own form. The form lives under the product in the dashboard: Products → Surge XT → Form Builder

Why it matters for developers: submitted form data is retrievable via the Polaris API, visible on each license's detail page in the dashboard, and can trigger a webhook to your own server on each activation.

1 Two Modes — Choose One

Option 1
GUI Template
Polaris renders a hosted activation form inside the plugin. Developer picks a visual template, toggles which fields appear, adds their branding. One line of JUCE code — openActivationForm().
Option 2
API-Driven
Developer defines a JSON Schema field-by-field. The JUCE SDK fetches the schema and the developer renders their own native plugin UI. Full visual control, more code required.
Rule of thumb: Most indie plugin devs will choose GUI Template. API-driven is for studios with a dedicated UI dev who wants pixel-perfect control.

2 GUI Template — Click-Through Steps

1
Choose Type
Click the GUI Template card (left option). It highlights with a teal border. Click Next → to continue.
2
Pick a Template
Choose from 4 visual styles: Dark Minimal, Light Clean, Warm Dark, Compact. Click any thumbnail to select it. Dark Minimal matches the Polaris brand. Compact is designed for small plugin windows.
↳ In production: branding (logo, colours) will be applied automatically from your account settings.
3
Configure the Form
Left column: toggle individual fields on/off. Full Name, Email, License Key are on by default. Company and Phone are off. License Key is locked — it's always required. Right column: company logo (auto from settings), success screen copy, GDPR consent toggle, webhook URL.
↳ Try toggling GDPR and Webhooks — each section collapses/expands. All inputs are editable.
4
Preview & Publish
Live preview of the form as it appears in the plugin window. Fill in the fields and click Activate Plugin → to see the success screen. Click Save & Publish to mark the form live.

3 API-Driven — Click-Through Steps

1
Choose Type
Click the API-Driven card (right option). Click Next → — this skips directly to the schema builder, bypassing template selection.
2
Build Schema — 3-Panel Builder
Left panel: field library. Standard types (Text, Email, Number, Select…) plus Polaris types (License Key, Machine ID, Plugin Host, OS Version). Drag or click to add.

Centre panel: form canvas. Fields listed in order with drag handles. Click any field card to select it. Sub-tabs switch between Builder view, JSON Schema output, and API Docs.

Right panel: field config. Label, Field ID code, type, placeholder, required toggle, validation rules.
↳ Click each field card — the right panel updates. Try the JSON Schema and API Docs tabs in the centre.
3
Save & Publish
Once fields are configured, click Save & Publish. The schema is served at GET /api/v1/products/{slug}/form. The JUCE SDK fetches this endpoint and the dev renders their own UI from it.

4 Field IDs — The Key Concept

Every form field has a Field ID — a snake_case code identifier. This is how submitted data is stored against the license and how the JUCE SDK references each field. It appears in the JSON Schema and in the license detail page.

user_full_nameFull Namestring
user_emailEmail Addressemail
activation_codeLicense Keylicense-key · polaris

Important: once a form is published and users have activated, renaming a Field ID breaks the mapping — existing license records store data under the old key. The UI warns developers about this.

5 Other Screens in the Mockup

Analytics Tab

Switch to the Analytics tab in the top product nav. Shows total activations, completion rate, average time to fill, field completion rates (bar chart), DAW breakdown, and a recent submissions table.

License Detail — Form Responses

Use the screen switcher at the very top of the mockup to switch to Screen 2 — License Detail. Scroll down to see the Form Responses card — this shows submitted data for that license, keyed by Field ID, including GDPR consent.

JSON Schema Tab

Switch to the JSON Schema tab. Shows the W3C Draft 2020-12 compliant schema output, including x-polaris-type extensions for Polaris-specific fields.

? Open Questions & Discussion Points

These are the questions likely to come up in review. Some have a proposed answer; some need a decision from the team.

Form versioning — what happens when a developer edits a published form?▾

The problem: if a dev removes a field after 500 users have activated, those 500 license records have data under a field ID that no longer exists in the current schema. If they add a field, new activations have it, old ones don't.

Proposed approach: introduce a Draft / Published state. Edits happen in Draft. Publishing creates a new version (v1, v2…). Each license record stores which schema version it was submitted against. The dashboard shows the form version alongside Form Responses.

Decision needed: can a dev have two active versions simultaneously (e.g. for existing users re-activating on a new machine using an old key)? Or is the latest published version always the one used?

Field ID locking — should field IDs be locked after publishing?▾

Currently the mockup warns developers but doesn't prevent them from renaming a Field ID post-publish. Two options:

  • Warn only (current): editable with a yellow warning. Dev accepts responsibility.
  • Hard lock: Field ID becomes read-only once the form is published. Dev can add new fields but cannot rename existing ones. If they need a new ID, they add a new field and deprecate the old one.

Hard lock is safer for data integrity. Recommend this approach.

Form analytics — what metrics matter most to plugin developers?▾

The mockup includes: total activations, completion rate, avg fill time, field completion rates, DAW breakdown, recent submissions table.

Additional metrics worth discussing:

  • Activation-to-purchase lag — time between license minted and first activation
  • Re-activation rate — how often users activate on a second machine
  • Failed activations — invalid license key entry rate
  • Geographic breakdown — country from IP at activation time

Decision needed: is analytics a Phase 1 feature or Phase 2?

GDPR & data privacy — who owns the collected data?▾

The form collects personal data (name, email) on behalf of the plugin developer. This creates a data controller / processor relationship. Key questions:

  • Is Polaris the data processor and the plugin developer the controller? (Most likely yes.)
  • Does the developer need to provide their own Privacy Policy URL in the consent checkbox? (Yes — mockup already has this field.)
  • Can an end user request deletion of their form data? If so, Polaris needs a deletion API and the dashboard needs a "Delete form data" action on each license.
  • Data retention: how long is form data kept after a license is revoked?

Recommend: legal review before shipping this feature publicly.

Webhooks — what happens if the webhook endpoint is down?▾

If the developer's webhook URL returns an error or times out, the activation should still succeed — the webhook is a notification, not part of the activation flow.

Proposed behaviour: retry up to 3 times with exponential backoff (30s, 5min, 30min). Failed deliveries are logged and visible in the dashboard. Developer can replay a failed webhook from the license detail page.

Decision needed: is webhook delivery logging a Phase 1 requirement?

Re-activation — does the user fill the form again on a second machine?▾

Two options:

  • Form once per license: on second activation, Polaris skips the form (or pre-fills from stored data). The machine_name and plugin_host fields are still captured for the new activation record.
  • Form every activation: user fills it each time. Creates a separate form data record per activation, which is richer but adds friction.

Recommend: form once per license, Polaris types (machine_name, plugin_host) captured silently on every activation regardless.

Offline activation — what if the plugin launches without internet?▾

Both modes require an internet connection at activation time — the JUCE SDK needs to reach the Polaris API to validate the license key and store form data.

Options for offline users:

  • Offline grace period: allow the plugin to run for N days before requiring activation (e.g. 7 days). After that, requires online activation.
  • Machine-locked offline file: Polaris generates a signed activation file the user can transfer manually. More complex to implement.

This is likely out of scope for Form Builder Phase 1 but worth flagging to the SDK team.

Template customisation depth — can developers go beyond field toggles?▾

The current mockup allows: choosing a template style, toggling fields on/off, reordering fields, setting the company logo (from account settings), and editing success screen copy.

Developers may also want:

  • Custom accent colour override (beyond the 4 template presets)
  • Custom font to match their plugin brand
  • Custom form width (for different plugin window sizes)
  • Adding a custom header image or illustration

Recommendation: ship with the 4 presets + logo/branding from settings. Custom colour/font as a Phase 2 "Advanced Branding" feature.

Multiple forms per product — activation form vs. registration form?▾

Currently one form per product. A developer might want:

  • An activation form — shown on first launch, collects registration data
  • A support form — collects bug reports or feedback from within the plugin
  • An upgrade form — shown when a user upgrades from v1 to v2

This is a meaningful product expansion but adds significant complexity (form targeting, triggers, multiple form data records per license). Recommend: single form per product for v1. Log this as a roadmap item.

API-driven mode — does Polaris validate the submitted data server-side?▾

Yes. When the JUCE SDK calls activateLicense(licenseKey, formData), Polaris runs the submitted data through the published JSON Schema using Ajv (server-side validation) before storing it.

If validation fails (e.g. email format invalid, required field missing), the activation is rejected and the SDK receives an error response with the validation errors. The developer's UI should surface these to the user.

In GUI Template mode, client-side validation in the hosted form catches errors before submission.
